# WhatsAppNode

Сервер інтеграції WhatsApp (Baileys) з CRM (ASP.NET webhook).

## Актуальний стан (оновлено 2026-04-09)

### Що вже реалізовано

1. Стабілізовано обробку `connection.close` у Baileys:
   - додано логування **коду + причини** від'єднання;
   - для тимчасових кодів (`restartRequired`, `connectionClosed`) залишено auto-reconnect;
   - для `401 (loggedOut)` — очищення сесії та повідомлення CRM.
2. Додано окрему обробку для `код 405`:
   - примусове очищення локального `auth-state`;
   - автоматичний повторний старт реєстрації через 5 секунд;
   - retry запускається навіть якщо `lineId` тимчасово відсутній (щоб QR не "зависав").
3. Додано централізований планувальник retry (`scheduleSessionRetry`) з логуванням причини повторної спроби.
4. Оновлено документацію API та блок діагностики 405 на основі логів від **09.04.2026**.

## API

### `POST /registerwhatsapp`
Реєстрація/ініціалізація WhatsApp-сесії.

Body:
```json
{
  "phone": "+380XXXXXXXXX",
  "lineId": "crm-line-id"
}
```

### `POST /sendmsg`
Відправка повідомлення в WhatsApp.

Body:
```json
{
  "from": "+380XXXXXXXXX",
  "to": "+380YYYYYYYYY",
  "message": "Текст повідомлення",
  "contentType": "text"
}
```

Підтримувані `contentType`:
- `text`
- `image` (за наявності `filePath`)
- `document` (за наявності `filePath`)

### `DELETE /sessiondelete/:phone`
Видаляє сесію номера.

### `GET /whatsapp_health`
Повертає health-статуси активних сесій у форматі:
```json
[
  {
    "phone": "+380XXXXXXXXX",
    "status": 1
  }
]
```
де `status: 1` — сесія активна, `status: 0` — неактивна/неавторизована.

## Чому виникає `Код: 405` при реєстрації

### Що видно у ваших логах (09.04.2026)

Node:
- `20:15:58` — старт створення сесії для `+380632910991`.
- `20:16:01` — `connection close`, `Код: 405`, `Connection Failure`.
- `20:16:01` — очищення auth-state.
- `20:16:02` — завершення клієнта та видалення директорії `auth_sessions/session-380632910991`.

CRM (prog.top):
- `20:15:34` — "У базі даних немає зареєстрованих номерів".
- `20:15:34` — "Обробка нової реєстрації з QR-кодом для номера +380632910991".

### Що це означає

Помилка `405` тут приходить на етапі встановлення/відновлення WhatsApp Web-сесії. Для цього проєкту це типово означає, що поточний локальний auth-state для номера невалідний або сесія була розірвана на стороні WhatsApp, тому потрібна повна повторна QR-авторизація.

### Що вже робить сервер автоматично

1. Ловить `405` в `connection.close`.
2. Видаляє локальну сесію (`auth_sessions/session-...`).
3. Запускає повторну ініціалізацію через 5 секунд.

### Що зробити вручну (чекліст)

1. Викликати `DELETE /sessiondelete/:phone` для проблемного номера.
2. Перезапустити Node-процес (очистити in-memory стани `clients`, `qrTimers`, `sessionStatus`).
3. Повторно викликати `POST /registerwhatsapp` **з обома полями**:
   ```json
   {
     "phone": "+380632910991",
     "lineId": "your-line-id-from-crm"
   }
   ```
4. Відсканувати новий QR з основного акаунта WhatsApp на телефоні.
5. Перевірити `GET /whatsapp_health` — очікується `status: 1`.

### Якщо 405 повторюється навіть після нового QR

1. Перевірити, чи номер не має обмежень/бану в WhatsApp.
2. Перевірити стабільність мережі/проксі/VPN між сервером і WhatsApp Web.
3. Оновити залежності (`@whiskeysockets/baileys`) і повторити реєстрацію на чистій сесії.
4. Переконатись, що в один момент часу для номера не працюють паралельно дві сесії (два процеси Node).

1. Додати валідацію вхідних payload (`zod`/`joi`) для `/registerwhatsapp` і `/sendmsg`.

   - інтеграційні тести endpoint-ів через `supertest`.
3. Додати rate limit на API endpoint-и.
4. Додати structured-логування та кореляційний `requestId` між Node та CRM.
5. Додати endpoint для ручної ресинхронізації конкретної сесії без перезапуску процесу.
6. Додати production-документацію запуску (`pm2/systemd`, автоперезапуск, ротація логів).
7. Додати базову авторизацію/секрет для API endpoint-ів.
